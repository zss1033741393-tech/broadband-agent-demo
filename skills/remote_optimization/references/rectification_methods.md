# 整改方式编号对照表

本表由 `remote_optimization` Skill 脚本在组装 CLI 参数时参考，同时供 PlanningAgent 生成"远程闭环处置方案"段落时选择合适的组合。

## 编号 → 动作

| 编号 | 动作 | 适用场景 | 业务风险 |
|---|---|---|---|
| `1` | 设备重启 | 网关/ONU 响应异常、链路不稳 | 短暂断网（约 1-2 分钟） |
| `2` | 信道切换 | 2.4G/5G 信道干扰严重 | 已连接 STA 可能短暂断线重连 |
| `3` | 2.4G 功率调整 | 2.4G 覆盖弱或边缘信号不足 | 相邻 AP 干扰可能变化 |
| `4` | 5G 功率调整 | 5G 覆盖弱或吞吐不达标 | 相邻 AP 干扰可能变化 |

## 常用组合

| 场景 | 建议编号 | 说明 |
|---|---|---|
| 单用户卡顿、原因不明 | `[1, 2]` | 先重启 + 切换信道，最小影响 |
| 卖场走播（走动覆盖弱） | `[2, 3, 4]` | 信道 + 功率全调，不重启避免业务中断 |
| 楼宇直播（静态场景，初次保障） | `[1, 2, 3, 4]` | 全部整改 |
| 紧急投诉处置 | `[1, 2, 3, 4]` | 全部整改 + `strategy=immediate` |
| 常规维护（闲时） | 不传 / `null` | 全部整改 + `strategy=idle` |
| 仅验证信道可用性 | `[2]` | 单项最小动作 |

## 约束

- 编号列表里只能出现 `1 / 2 / 3 / 4`，其他值会被 FAE 平台拒绝
- 不传 / `null` / `[]` 在脚本层统一视为"全部整改"，不会传 `--rectification-method` 参数给 CLI
- 组合中出现 `1`（设备重启）时，业务高峰时段建议搭配 `strategy=idle`，避免直接中断业务
